## 1. Vision

AI Autopilot transforms PiCast from a manual media player into a **personal TV channel**. When enabled, an AI "DJ" continuously selects and queues content based on your taste profile, time of day (via PiPulse blocks), and viewing history. It's an always-on, silent DJ that just plays what you want — no interaction required.

### Core Principles

1. **Always-on DJ** — Not on-demand recommendations. Continuous, seamless content flow.
2. **Silent DJ** — No personality, no explanations unless inspected. Just great picks.
3. **Two-tier brain** — Opus 4.6 generates daily rulesets; deterministic Python scripts on Pi execute them.
4. **Respect manual control** — AI never overrides explicit user choices on other devices.
5. **Cost-aware** — ~1 Opus API call/day for taste profile refresh. Zero cost during playback.

## 2. Design Decisions (From Interview)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| AI brain location | Hybrid: Mac-orchestrated + Pi execution | Pi too weak for LLM; Opus generates patterns, Pi follows rules |
| Autopilot UX | Always-on DJ | Continuous queueing like a personal TV channel |
| Multi-device | Two modes (Single + Fleet) | Single = AI on this PiCast only; Fleet = AI controls all Pi Zeros |
| Fleet routing | AI decides everything | Autonomously picks what plays where based on taste + device context |
| Content discovery | Pool-only toggle + YouTube enhanced search | Twitch live streams as additional source |
| YouTube recs | Skip for v1 | Start with enhanced search only |
| Twitch | Live streams only | Find matching live channels, switch when offline |
| Mood detection | PiPulse blocks | Use existing time blocks for context (morning, deep work, evening) |
| Autopilot UI | Dedicated view | New page showing AI state, upcoming picks, feedback controls |
| Queue depth | 3-5 ahead | Balance between responsiveness and seamless playback |
| Taste refresh | Daily | Opus analyzes yesterday's data, generates fresh strategy each morning |
| AI personality | Silent | No character, no comments. Content speaks for itself |
| Discovery Agent | Bundled into Autopilot | AI-powered search replaces the never-shipped basic keyword discovery |
| v0.8.0 cleanup | Do first | Lint + TUI tests + URL pre-validation before adding AI complexity |

```

                    │   Opus 4.6 Analyzer      │
                    │                           │
                    │   Inputs:                 │

                    │   - Rating changes         │
                    │   - Skip patterns          │
                    │   - Block schedule          │
                    │   - Pool inventory          │
                    │                           │
                    │   Outputs:                │
                    │   - taste-profile.json     │
                    │   - block-strategies.json  │
                    │   - discovery-queries.json  │
                    │   - content-scores.json     │
```

### 3.3 Content Sources

| Source | Mode | Implementation |
|--------|------|---------------|
| Autoplay Pools | Pool-only toggle ON | Existing weighted random, upgraded with AI scoring |

### 3.4 Multi-Device Architecture

```
MODE 1: Single PiCast                MODE 2: Fleet
┌──────────────┐                     ┌──────────────┐
│   PiCast     │                     │   PiCast     │ (Brain)
│   + Autopilot│                     │   + Autopilot│
│   (self)     │                     │   + Fleet Mgr│
└──────────────┘                     └──┬───┬───┬───┘
                                        │   │   │
                                   ┌────▼┐ ┌▼──┐┌▼────┐
                                   │Zero1│ │Z2 ││Z3   │
                                   │Bed  │ │Off││Liv  │
                                   └─────┘ └───┘└─────┘
```

**Fleet Mode Details:**
- PiCast (main unit) is the brain; Pi Zeros are receivers
- Brain discovers all PiCasts via existing mDNS/Zeroconf
- Brain pushes content to Zeros via their REST API (`/api/play`, `/api/queue/add`)
- Each Zero has a device profile (room, purpose) stored in PiCast config
- AI decides what content goes to which device based on device profiles + time
- **Override protection:** If a device has manual content playing (not from autopilot), AI skips it

## 5. Data Flow

### 5.1 Daily Refresh Cycle

```
1. [06:00] Mac cron/launchd triggers refresh script
2. Script pulls from PiCast:
   - GET /api/autoplay/history?since=yesterday
   - GET /api/autoplay/pool (all pools with ratings)
   - GET /api/status (current block schedule)
3. Script calls Opus 4.6 with structured prompt:
   - "Given this viewing data, generate taste profile and strategies"
   - Input: ~500 tokens (structured JSON)
   - Output: ~1000 tokens (taste-profile.json)
   - Cost: ~$0.02/day
4. Script pushes results to PiCast:
   - rsync taste-profile.json to Pi
   - OR: POST /api/autopilot/profile (new endpoint)
5. Pi Autopilot Engine reloads profile
```

### 5.2 Continuous Playback Loop (On Pi)

```
while autopilot_enabled:
    1. Check current PiPulse block
    2. Load block strategy from taste-profile.json
    3. If queue_depth < target (3-5):
       a. If pool_only_mode:
          - Select from pool using AI-enhanced scoring
       b. Else:
          - 70% pool picks, 30% discovery picks (configurable)
          - Discovery: run yt-dlp search with Opus-generated queries
          - Filter by duration, already-played, skip history
    4. Add selected video to queue via /api/queue/add
    5. Wait for playback events (completion, skip, manual override)
    6. Update local metrics (skip_count, completion_count)
    7. If block changed (PiPulse webhook): reload strategy
```

### 5.3 Fleet Coordination Loop

```
while autopilot_enabled and fleet_mode:
    1. Poll all known devices: GET /api/status
    2. For each idle device (nothing playing, no manual override):
       a. Select content using device profile + block strategy
       b. POST /api/queue/add to that device
    3. For each device playing autopilot content:
       a. Monitor completion via SSE or polling
       b. Queue next item when 1 item remaining
    4. For devices with manual content:
       a. Skip — user is in control
       b. Resume autopilot when manual content ends (if still enabled)
```

## 6. New Database Tables

```sql
-- AI taste profile storage (synced from Mac daily)
CREATE TABLE IF NOT EXISTS autopilot_profile (
    id INTEGER PRIMARY KEY DEFAULT 1,
    profile_json TEXT NOT NULL,
    generated_at TEXT NOT NULL,
    loaded_at TEXT DEFAULT (datetime('now')),
    version INTEGER NOT NULL DEFAULT 1
);

-- Fleet device profiles
CREATE TABLE IF NOT EXISTS autopilot_devices (
    device_id TEXT PRIMARY KEY,          -- mDNS name or config key
    room_name TEXT,                       -- "bedroom", "office", "living"
    device_type TEXT DEFAULT 'single',   -- "single" or "fleet_member"
    last_seen TEXT,
    is_manual_override BOOLEAN DEFAULT 0
);

-- Autopilot session log (what AI decided and why)
CREATE TABLE IF NOT EXISTS autopilot_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT DEFAULT (datetime('now')),
    action TEXT NOT NULL,                 -- "queued", "discovered", "skipped_device"
    video_id TEXT,
    device_id TEXT,
    block_name TEXT,
    source TEXT,                          -- "pool", "youtube_search", "twitch_live"
    score REAL,                           -- AI confidence score
    reason TEXT                           -- brief explanation for debugging
);
```

## 7. New API Endpoints

```
# Autopilot control
POST   /api/autopilot/toggle              # Enable/disable autopilot

# Taste profile

# Source toggles
POST   /api/autopilot/sources             # Enable/disable: pool, youtube, twitch

# Fleet
GET    /api/autopilot/fleet               # Fleet device status
POST   /api/autopilot/fleet/device        # Register/update device profile

# Feedback
POST   /api/autopilot/feedback            # "more like this" signal
```

## 9. Configuration

New `[autopilot]` section in `picast.toml`:

```toml
[autopilot]
enabled = false
mode = "single"                          # "single" or "fleet"
queue_depth = 4                          # target items ahead
pool_only = false                        # restrict to existing pools
discovery_ratio = 0.3                    # % of picks from discovery vs pool
twitch_enabled = false
twitch_categories = ["Music", "Art"]

[autopilot.refresh]
method = "mac"                           # "mac" (rsync) or "api" (POST profile)
mac_host = "10.0.0.1"                   # Mac IP for pulling data
schedule = "daily"                       # "daily", "weekly", "manual"

[autopilot.fleet]
# Device profiles for fleet mode
[autopilot.fleet.devices.bedroom]
host = "picast-bedroom.local"

mood = "chill"

[autopilot.fleet.devices.office]
```

## 13. Open Questions for Plan Mode

These will be explored with clarifying questions in the plan mode session:

1. **Local model feasibility:** Can a small model (TinyLlama, Phi-3-mini) on Pi follow Opus-generated rule files? What format should the rules be in for a dumb executor?
2. **Twitch auth:** Does Twitch require API keys for browse/search? Or can we use unauthenticated endpoints?
3. **Fleet discovery:** Should fleet mode use Zeroconf (already exists) or require explicit config?
4. **Profile format:** Is JSON the right format, or should Opus generate Python code/lambda functions directly?
5. **Fallback behavior:** What happens if the taste profile is stale (>48h old)? Fall back to weighted random?
6. **Session persistence:** When autopilot is toggled off then on, should it resume the previous strategy or start fresh?
